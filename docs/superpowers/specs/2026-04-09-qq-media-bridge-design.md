# QQ 富媒体双向桥接设计

## 目标

在 `qq-codex-bridge` 中一次性补齐 QQ 富媒体双向能力：

- QQ -> Codex：接收图片、文件、语音、视频
- Codex -> QQ：发送图片、文件、语音、视频
- Codex 出站同时支持 `<qqmedia>路径或URL</qqmedia>` 和 Markdown 声明
- QQ 入站媒体既保留本地路径，也尽量提取成可读文本注入 Codex 线程

## 约束

- 保持现有 QQ gateway WebSocket 入站主路径不变
- 保持按 QQ 私聊/群聊隔离的 SQLite 会话模型不变
- 富媒体能力优先围绕当前项目真实可控能力实现，不照搬 `openclaw-qqbot` 中尚未接入本仓库的能力
- 语音转写、文件正文抽取、图片尺寸/摘要等提取能力允许先做“能提取则提取，不能提取则保底给路径和元信息”
- 文本消息和媒体消息必须能在同一轮回复中共存

## 设计

设计上引入统一的 `media artifact` 模型，作为 QQ 入站附件、Codex 上下文注入和 QQ 出站发送之间的公共表示。每个 artifact 至少包含 `kind`、`sourceUrl`、`localPath`、`mimeType`、`fileSize`、`originalName` 和可选的 `extractedText`。这样图片、语音、视频、文件都能走一套一致的下载、持久化、注入和发送流程。

QQ 入站侧需要从 gateway 事件中识别附件，下载到本地媒体目录，并按类型生成上下文。进入 Codex 的消息不只保留文字内容，还会拼接“附件清单 + 本地路径 + 可提取文本/摘要”。这满足“路径可见”和“尽量可读”两个目标：图片至少有本地路径和尺寸信息，语音尽量带转写，文本类文件尽量带正文摘要，视频至少有路径和元信息。

Codex 出站侧增加媒体声明解析层，同时识别 `<qqmedia>...</qqmedia>` 和 Markdown 中的图片/媒体引用。解析结果统一交给 QQ 出站媒体路由器，由它按类型调用 QQ 对应发送接口。文本与媒体混发时，保持声明顺序发送；超出大小限制时，明确告知用户具体哪个文件无法发送，而不是静默失败。

## 验证

- 合同测试覆盖附件归一化、媒体声明解析和 QQ 媒体发送路由
- 单元测试覆盖媒体工件模型、文件类型识别、提取结果拼接和大小限制处理
- 全量 `npm test` 与 `npm run check`
- 真实 `pnpm dev` 联调，验证 QQ 发图片/文件/语音/视频到 Codex，以及 Codex 用 `<qqmedia>` 或 Markdown 回发媒体到 QQ
